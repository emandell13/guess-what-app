Mobile redesign — one direction, across game states

Solves the three big pain points: (1) question scrolling out of view, (2) keyboard covering the answer grid, (3) strikes and hint hidden while guessing. Layout is now a three-zone shell: sticky header (logo / question / strikes / hint) → scrollable answer board → sticky input. End-game uses a bottom sheet instead of a centered modal.

M · Anti-pattern
What we're avoiding: if the sheet stays at its medium detent, the keyboard covers the submit button. Today's Bootstrap modal has this exact bug.

How the detent swap works

  1. Rest state: sheet opens at its "medium" detent (~60% viewport), letting the dimmed game board show through.
  2. Input focus: the vote input's focus handler (or visualViewport.resize listener) triggers the sheet to animate to its "tall" detent — top: env(safe-area-inset-top) + 44px, bottom: visualViewport.height distance from the real viewport bottom.
  3. Internal layout: sheet uses flex-column. Header (title, subtitle, progress) is a scrollable middle region; input + submit live in a flex-shrink:0 footer pinned to the bottom of the sheet.
  4. Blur: on input blur, sheet animates back to medium detent. User can also swipe the grab handle down to dismiss, same as any step.

This is the same pattern iOS Messages and Safari's own sheets use — the user doesn't have to scroll to find the submit button because it's always one keyboard-row away from their thumb.

Design notes — what changes and why

The three-zone shell

Today the page is one scrolling column, which means on mobile the question disappears above the fold the moment the keyboard opens. The redesign pins three zones:

  • Top (sticky): mini-logo, date, question, strike pips, hint chip. Compact enough to survive a keyboard-open viewport (~360px tall).
  • Middle (scrollable): the 5 answer tiles. If content overflows, only this region scrolls.
  • Bottom (sticky, safe-area aware): the guess input + host feedback bubble. Uses position: fixed + visualViewport APIs so it always sits just above the keyboard.

Strike/hint row, always on

Currently the strikes counter and hint button are mid-page and scroll away. Moving them into the sticky header as a pip meter + chip (strikes shown as 5 dots rather than a "Strikes: X/5" text) keeps state visible without taking a full row of height.

Host commentary → inline bubble

The desktop floating host bubble gets obscured on mobile. We swap it for an inline bubble anchored above the input — same avatar, same voice, always visible. It auto-dismisses on next keystroke or after 4s.

Answer tiles, tightened

Reduced min-height (44→48px), smaller rank font, and a points badge on the right. All 5 tiles fit in ~320px of vertical space so the full board stays visible with the keyboard open on a 667-tall phone.

End-game: modal → bottom sheet

Bootstrap modals feel foreign on mobile and the vote step (with its own input) is fragile. A bottom sheet with a grab handle, 4-segment progress bar, and native-feeling swipe dismiss is the right primitive. Each of the four steps fits within 78% of viewport height so nothing is ever clipped.

New-ish additions to consider

  • Share-result CTA in the final step (Wordle-style emoji grid) — drives organic growth.
  • Daily reminder toggle using the Web Notifications / calendar-link pattern.
  • Haptics on correct guess / strike (iOS Safari supports navigator.vibrate in PWA context).
  • Safe-area insets respected everywhere (env(safe-area-inset-bottom)) so input doesn't sit under the home indicator.

Implementation sketch

  • New wrapper: <div class="app-shell"> with display:flex; flex-direction:column; height:100dvh.
  • Header: existing content re-laid, stickied.
  • Board: existing #answer-boxes container, wrapped in scroll region.
  • Input: existing #guess-form-container, given position:sticky; bottom:0 plus a visualViewport listener to push it above the keyboard on iOS.
  • End-game modal: rebuild the Bootstrap modal as a bottom sheet component (or swap in a lightweight lib like vaul). Reuse the existing 4-step internal logic.
  • Desktop remains unaffected — the shell only activates below a breakpoint (e.g. @media (max-width: 768px)).
